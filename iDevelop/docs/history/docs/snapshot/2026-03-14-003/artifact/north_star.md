# North Star

## Human Goal

Codex での開発時に、文書管理と結果整理を一か所で扱える軽量ダッシュボードを作る。
利用者は文書検索と編集、保持データと結果データの閲覧・集計・図表表示、可能であればダッシュボードからの遠隔コード参照とデバッグを行える。

## Scope Boundary

- このプロジェクトの対象は `iDevelop/` 配下のダッシュボード設計準備と実装である。
- メインプロジェクト本体の実装や規約本文は対象外であり、必要なら参照だけ行う。
- ダッシュボードは Codex による継続的カスタマイズを前提とする。

## Non-Negotiables

- 軽量であることを優先し、重い依存や不要な常駐処理を安易に持ち込まない。
- 拡張にオープンで、既存コア修正にはクローズな設計を採る。
- MVC の責務分離を崩さない。
- BDD で振る舞いを定義し、TDD で実装を進める。
- 文書画面とデータ画面は初期スコープに含める。
- コード参照とデバッグ画面は optional scope として扱い、無理に最初から固定しない。
- ダッシュボード関連の文書、コード、履歴は `iDevelop/` 配下に閉じる。
- companion project `iSensorium/` には境界ルールだけを残し、ダッシュボード固有事項は持ち込まない。
- このセッションでは実装開始条件の固定を優先し、実コード着手は次セッションへ委譲する。

## Invariants

### Fixed

- 利用主体は Codex と人間の共同開発者である。
- カスタマイズポイントは追加しやすく、既存画面の意味は壊しにくい構造を保つ。
- 文書管理とデータ可視化は同一ダッシュボードから到達できる。

### Hypothesis

- 初期構成はローカル実行の Web ダッシュボードが最も扱いやすい可能性が高い。
- コード参照とデバッグは初期段階では read-only 寄りの機能から始める方が安全である。
- 図表表示は汎用チャート層を先に作る方が拡張性を保ちやすい。
- 初回の実装セッションでは、文書画面とデータ画面の must scope から入るのが最も安全である。

### Disposable

- 採用フレームワーク
- 初期 UI テーマ
- 画面分割の細部
- データ保持形式の細部

## Success Criteria For This Thread

- サブプロジェクトの分離境界が明文化されている。
- ダッシュボードの初期スコープが文書画面、データ画面、optional なコード画面として整理されている。
- MVC、BDD、TDD を土台とする設計原則が固定されている。
- 次セッションで新規に実開発を開始できる。
